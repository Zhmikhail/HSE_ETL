# Мини‑проект про ETL и Airflow

Это мой финальный мини‑проект по теме ETL / репликация и аналитические витрины.  
Тут я просто показываю, как беру данные из MongoDB, перегоняю их в Postgres через Airflow и собираю пару понятных витрин для аналитики. Ничего супер сложного, но всё работает как надо (ну, по крайней мере должо работать).

### Что тут вообще есть

- **MongoDB** — база `etl_final` с коллекциями:
  - `UserSessions`, `EventLogs`, `SupportTickets`, `UserRecommendations`, `ModerationQueue`.
- **PostgreSQL** — база `etl_final_warehouse` с двумя схемами:
  - `raw` — почти сырые, но уже чутка почищенные данные из Mongo (типы, даты, дубли и т.д.).
  - `marts` — финальные аналитические витрины, по которым уже можно строить отчётики.
- **Генерация данных** — скрипт `mongo_seed.py`, который забивает Mongo тестовыми данными, шоб было с чем работать.
- **Airflow** — два DAG-а:
  - `mongo_to_postgres_replication` — тащит данные из Mongo, приводит их к вменяемому виду и грузит в `raw.*` в Postgres.
  - `analytics_marts` — на основе `raw.*` собирает витрины в `marts.*`.

### Как устроен первый DAG — `mongo_to_postgres_replication`

Сначала создаётся схема `raw` и нужные таблицы в Postgres, а дальше для каждой коллекции из Mongo идёт цепочка extract → transform → load.

По смыслу оно делает следующее:

- Для `UserSessions`:
  - правит форматы дат на TIMESTAMPTZ;
  - считает длительность сессии;
  - режет дубли;
  - пишет всё это в `raw.user_sessions` с upsert по `session_id`.
- Для `EventLogs`:
  - приводит дату события;
  - нормализует JSON‑поле `details`;
  - убирает дубли;
  - пишет в `raw.event_logs` с upsert по `event_id`.
- Для `SupportTickets`:
  - приводит даты;
  - считает `resolution_time_seconds` и число сообщений;
  - убирает повторы;
  - пишет в `raw.support_tickets` с upsert по `ticket_id`.
- Для `UserRecommendations`:
  - приводит дату `last_updated`;
  - нормализует список рекомендованных товаров;
  - пишет в `raw.user_recommendations` с upsert по `user_id`.
- Для `ModerationQueue`:
  - приводит дату `submitted_at`;
  - нормализует список флагов;
  - чистит дубли;
  - пишет в `raw.moderation_queue` с upsert по `review_id`.

### Второй DAG — `analytics_marts`

Тут уже чистая аналитика:

- Создаётся (если её ещё нет) таблица `marts.user_activity_mart`.
- В неё по каждому пользователю и дате считается:
  - сколько было сессий;
  - средняя длительность сессий;
  - сколько было действий;
  - сколько уникальных страниц он потрогал.
- Аналогично создаётся `marts.support_efficiency_mart`.
- Туда по дате, типу проблемы и статусу тикета считается:
  - сколько тикетов;
  - среднее время решения;
  - сколько тикетов остаётся открытыми.

### Что в итоге получается

В `raw` лежит по сути аккуратный слепок данных из MongoDB, но уже без мусора: нормальные типы, даты, почищены дубликаты.  
В `marts` — две витрины: одна про активность пользователей, вторая про эффективность поддержки. Их уже можно крутить в BI, делать дашборды или просто руками смотреть, как живёт сервис.

Проект сделан скорее как учебный пример: показать полный путь от Mongo до витрин в Postgres через Airflow, без лишней магии и оверинженеринга.
